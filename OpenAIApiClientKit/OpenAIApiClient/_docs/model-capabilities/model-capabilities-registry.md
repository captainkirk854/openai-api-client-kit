## Capability Taxonomy
The registry uses a consistent set of 16 weighted capability tags, grouped into four highРђЉlevel domains. Each tag uses a 0РђЊ5 scale, 
where `0` means unsupported and `5` indicates a primary strength.

### Core - *Fundamental cognitive and multimodal capabilities*
- `Reasoning` Рђћ depth, logic, multiРђЉstep problem solving
- `Text` Рђћ general text generation quality
- `Chat` Рђћ conversational alignment and instruction following
- `Vision` Рђћ ability to process images
- `AudioIn` Рђћ speech/audio input
- `AudioOut` Рђћ speech/audio output

### Advanced - *Higher-order behavioural and structured-output capabilities*
- `Critic` Рђћ ability to critique, evaluate, and refine content
- `Editor` Рђћ ability to rewrite, transform, or restructure content
- `JSONMode` Рђћ structured output reliability
- `Embedding` Рђћ vector embedding generation
- `ImageGeneration` Рђћ generative image capabilities

### Performance - *Speed, throughput, and model capability tier*
- `FastInference` Рђћ latency and throughput
- `HighPerformance` Рђћ overall capability tier

### Operational - *Cost, safety, and deployment characteristics*
- `LowCost` Рђћ costРђЉefficiency
- `Moderation` Рђћ safety and filtering strength
- `OpenWeight` Рђћ availability of openРђЉweight versions

## Registry JSON Schema
A strict JSON schema defines the structure of the registry. 

It enforces:
- Required capability groups
- Required tags within each group
- Integer weights between 0 and 5
- No unknown fields
- Extensibility for future model families

Each model entry follows this structure:
```
{
  "name": "model-name",
  "family": "model-family",
  "capabilities": {
    "core": { ... },
    "advanced": { ... },
    "performance": { ... },
    "operational": { ... }
  }
}
```
